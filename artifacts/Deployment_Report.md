# BlockCraft — Deployment Report

## Project Overview

**BlockCraft** is a pure-frontend, browser-native 3D voxel sandbox game. The production bundle is a set of static assets that can be served from any static host. There is **no backend, no database, no user accounts, and no third-party API calls** beyond a Google Fonts stylesheet (which can be self-hosted if needed).

**Version:** 1.0.0-beta  
**Bundle size:** ~1.03 MB JS (gzipped 291 KB), 20 KB CSS (gzipped 4 KB), 15.6 KB worker, 12-entry PWA precache (~1.05 MB total). Well under the 2.5 MB release gate.

## Environment Requirements

### Local Development
- **Node.js** ≥ 20 (tested with 24.x)
- **npm** ≥ 10 (tested with 11.x)
- A modern browser with **WebGL 2.0** and **Web Audio API** (Chrome 90+, Firefox 88+, Safari 14+, Edge 90+).
- For PWA testing: an HTTPS-capable local server (Vite preview runs HTTP only; use `vite preview --https` or a local tunnel for full PWA verification).

### Production
- Any static web host. **Cloudflare Pages** is the documented primary target.
- HTTPS is **required** for Service Workers (and therefore for offline PWA mode).
- No environment variables are required.
- No server-side runtime requirements.

## Required Environment Variables

**None.** The application has no backend and does not require any secrets, API keys, or runtime configuration.

If a Google Fonts CSP restriction is desired, host the Outfit/Inter fonts locally and remove the `<link>` from `index.html`. No code change required beyond that.

## Build Instructions

```bash
npm install            # one-time
npm run build          # outputs to dist/
```

The build runs `tsc --noEmit` first (type-check gate), then Vite produces:

- `dist/index.html`
- `dist/assets/index-<hash>.js` (1 MB)
- `dist/assets/index-<hash>.css` (20 KB)
- `dist/assets/chunkWorker-<hash>.js` (15.6 KB)
- `dist/manifest.webmanifest`
- `dist/sw.js` (service worker, generated by Workbox via vite-plugin-pwa)
- `dist/workbox-<hash>.js`
- `dist/favicon.svg`, `dist/icon-192.png`, `dist/icon-512.png`

The Service Worker registers itself automatically on first visit (see `src/main.tsx` `registerSW({ immediate: true })`).

## Validation Steps

After every build:

```bash
npm run typecheck      # 0 errors expected
npm test               # 126 / 126 passing
npm run build          # exits 0
```

Smoke tests before release:

1. Visit `http://localhost:4173` in a fresh browser profile.
2. Verify the **Main Menu** loads with the "Continue" button disabled (no save yet).
3. Click **New Game** → World Creation Screen renders.
4. Choose Easy / Medium (512×512) / leave seed blank → **Create World**.
5. World Generation Loading Overlay shows a progress bar and at least one stage message.
6. Game spawns the player on grass with the standard 4 wood logs in hotbar slot 1.
7. Walk forward (W), jump (Space), look around (mouse), break a block (left click), place a block (right click).
8. Press **E** to open the Inventory. Confirm:
   - Storage grid renders.
   - Crafting tab shows the 13 recipes, locked ones desaturated.
   - Drag and drop swaps slots.
9. Press **Escape** to open the Pause Menu. Adjust volume slider → confirm audio gain changes immediately.
10. Click **Backup World (.json)** → confirm `${worldName}.blockcraft` downloads.
11. Close the tab. Re-open. **Continue** button now reads "Resume: <world name>". Click it → world loads, player position preserved.
12. Open browser DevTools → Application → IndexedDB → `BlockCraftDB` → verify three stores (`worlds`, `inventories`, `chunk_deltas`) contain entries.
13. Hard refresh to verify the Service Worker activates (DevTools → Application → Service Workers → Status should be "activated and running").
14. Toggle airplane mode → reload → app should still load (PWA precache).
15. On a mobile viewport (Chrome DevTools device toolbar, iPhone 13), verify the virtual joystick and action buttons appear; flip handedness in Settings → joystick and action buttons swap sides.

## Docker Instructions

A Docker image is not required (the deliverable is a static bundle). However, if containerised hosting is preferred:

```dockerfile
# Dockerfile (reference)
FROM nginx:alpine
COPY dist/ /usr/share/nginx/html/
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

`nginx.conf` (snippet) — include SPA fallback to `index.html` and a long cache for hashed assets:

```nginx
server {
  listen 80;
  server_name _;
  root /usr/share/nginx/html;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  location ~* \.(js|css|woff2|webmanifest)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

Build and run:

```bash
docker build -t blockcraft:latest .
docker run --rm -p 8080:80 blockcraft:latest
# Open http://localhost:8080
```

## Production Deployment Steps (Cloudflare Pages)

### Option A — Connect Git (recommended)

1. Push the repository to GitHub / GitLab.
2. In Cloudflare dashboard → **Pages** → **Create application** → **Connect to Git**.
3. Select the repository.
4. Configure the build:
   - **Framework preset:** Vite
   - **Build command:** `npm run build`
   - **Build output directory:** `dist`
   - **Root directory:** `/`
   - **Node version:** 20 (set via `NODE_VERSION=20` environment variable on the Pages project)
5. Save and deploy. First build takes ~2 minutes.
6. Cloudflare auto-assigns a `*.blockcraft-xxx.pages.dev` URL. Optionally attach a custom domain.

### Option B — Direct Upload

```bash
npm install
npm run build
# Then use Wrangler:
npx wrangler pages deploy dist --project-name blockcraft
```

Or upload the `dist/` directory manually via the Cloudflare dashboard drag-and-drop.

### Verification After Deploy

- Visit the deployed URL over HTTPS.
- Open DevTools → Application → Manifest → confirm name "BlockCraft" and start_url "/".
- Lighthouse PWA audit should pass (installable + offline).
- IndexedDB persists across page reloads.

## Rollback Procedures

Cloudflare Pages keeps a full history of deployments.

1. Cloudflare dashboard → Pages → your project → **Deployments**.
2. Find the last-known-good deployment.
3. Click **⋯** → **Rollback to this deploy**.

For local or self-hosted deployments, redeploy the previous `dist/` artifact. Because everything is static and the IndexedDB schema is versioned, rollbacks are safe — the in-browser state machine reads `version` from the SaveBackupFormat and falls back gracefully on older clients.

## Backup Strategy

There is **no server-side data**. All game state is stored client-side in IndexedDB and LocalStorage. The user is responsible for their own backups.

Two mechanisms are provided:

1. **In-app export:** Pause Menu → "Backup World (.json)" downloads a portable `${worldName}.blockcraft` JSON file. This is the only safe way to migrate between browsers, devices, or platforms.
2. **Main Menu → Import Save** accepts any `.blockcraft` file and reconstructs the world in IndexedDB.

**Recommended user habit:** export a backup before closing the tab, especially on iOS Safari where the 7-day eviction policy may purge local data without notice. The in-game storage-guard warns at >80% quota usage.

**Operational backups:** none required at the host. Cloudflare Pages is content-distributed; the source of truth is the Git repository.

## Monitoring Recommendations

For Cloudflare Pages + a static SPA, the only first-party signal is Cloudflare Analytics (Web Analytics, free, no cookies). Add the snippet to `index.html` if desired — it does not break the "no telemetry" promise because it is opt-in and cookie-less.

**Server-side:** none — there is no server. The only host-side signal is Cloudflare's request log, which is enabled by default and inspectable in the dashboard.

**Client-side observability for development:** the in-game console exposes:
- `runStorageGuard()` → reports current storage estimate and warns if >80%.
- `getChunkWorker().dispose()` → tears down the Web Worker on demand.

**Crash detection:** unhandled errors in the Web Worker are surfaced via the `WorkerResponse.type === 'ERROR'` path and logged to the console. The render loop wraps the high-frequency tick in `requestAnimationFrame` and continues on transient GL errors.

## Logging Recommendations

- **Production:** rely on browser DevTools console for end-user debugging. No remote logging is included by default (consistent with the no-telemetry constraint).
- **During QA / dogfood:** set the engine's `console.warn`/`console.error` calls. The notable warning sites are:
  - `runStorageGuard` → "IndexedDB storage capacity running low"
  - `saveManager.runAutosave` → "autosave retry N" / "autosave failed"
  - `tierDetection` → no output (silent classification, surfaced only as a UI warning)
- **Structured log shape** (if a future remote-logging backend is added): the project uses simple string messages; adopting `pino` or `loglevel` is a clean future addition since the game has no current dependency on a logger.

## Operational Risks

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| iOS Safari evicts IndexedDB after 7 inactive days | High | Player progress loss | In-app storage-guard warning, manual `.blockcraft` export reminder, screen-spaced messaging in the Pause Menu |
| Tier 1 device attempts a Large (1024³) world | Medium | Tab OOM crash | World Creation screen soft warning modal per User-Flows §3.1, Tier 1 detection (navigator.deviceMemory, WEBGL_debug_renderer_info, UA) |
| Service Worker cache mismatch after deploy | Low | Stale assets | Workbox auto-update strategy (`registerType: 'autoUpdate'`) reloads clients on new SW activation |
| Cloudflare Pages outage | Low | Site unreachable | Static bundle can be re-hosted on any object storage; the Git repository is the source of truth |
| Block registry additions break saved worlds | Low | Import error on legacy save | Save format is versioned (`version: '1.0'`); future versions can branch with migration code in `importWorldSave` |
| WebGL context loss on tab backgrounding | Medium | Render freeze until re-focus | Three.js handles context restoration; the game loop resumes on the next `useFrame` tick |
